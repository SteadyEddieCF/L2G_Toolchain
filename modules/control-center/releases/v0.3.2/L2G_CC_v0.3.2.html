Documentation center

Operator guidance, governance rules, and the persistent release roadmap.

The L2G suite is a modular, local/offline advisory workflow. DocConverter normalizes sources, Scoper validates a draft boundary, Workshop facilitates review and remains the central coordination authority, Builder/Merger owns the formal workbook loop, and SSP owns SSP authoring and review. Control Center coordinates metadata and routes only; no stage may automatically create final CMMC conclusions.

Primary route: Workshop → SSP → Workshop. Workshop v71 generates l2g_ssp_handoff_v1 1.0; SSP v1.7 previews and selectively applies candidates, then returns l2g_ssp_return_package_v1 1.0. Workshop previews and selectively applies the returned authoring/review overlay. The optional l2g_ssp_round_trip_audit_v1 0.1 package is reduced to safe metadata and counters in Control Center observability. Control Center never edits SSP fields or creates assessment conclusions.

A current baseline is the expected module release recorded by this Control Center package. Loading a local manifest compares the reported module version, release focus, result, breaking-change flag, and contract metadata. The original manifest content is not persisted; only a minimal comparison summary may be saved in Control Center configuration.

Version drift means a loaded module manifest reports a release different from the v0.3.2 baseline. It is a review signal, not an automatic stable-contract rejection. Package compatibility uses package kind/version and bounded routing metadata. Wrong intended consumers or generator identities require review.

  • Use relative module paths inside one suite folder.
  • Do not use blob-based module launchers.
  • Browser file access and localStorage behavior can vary by browser and file origin.
  • Unknown package identity requires manual review; filenames alone are not compatibility proof.
  • Use generic, non-client-identifying filenames whenever practical. The active inspector may display the selected filename, but observability history persists only a redacted fingerprint-based label.
  • Do not load production CUI into ungoverned test environments.

Observability records metadata and safe counters only after explicit opt-in. Raw filenames are converted to fingerprint-based package labels before persistence. SSP audit change, import, and rejection bodies are never retained. Successful handoffs require separate operator confirmation, and missing optional audit evidence is never shown as a hard failure.

v0.3.2 closes the v0.3.1 review gaps without changing package authority or route meaning: raw filenames are not persisted, the authoritative Workshop v71 compatibility/release evidence and real round-trip fixtures are included in the deliverables, all six primary pages are browser-captured in both themes, and negative review_required/unsupported_version states are visually evidenced. The SSP audit contract remains intentionally limited to version 0.1 because that is the authoritative Workshop v71 contract.

Operator checklist

Standard end-to-end sequence

  1. Normalize fresh sources in DocConverter-L2G.
  2. Import Scope Context into L2G Scoper and validate the draft boundary.
  3. Import Intake, Scope Return, and Meeting Context into Workshop.
  4. Facilitate practice, evidence, provider, decision, and action review.
  5. Use Workshop ↔ Builder/Merger for the formal workbook loop.
  6. When SSP authoring context is mature, export SSP Handoff from Workshop.
  7. Preview and selectively apply candidates in SSP; export SSP Return Package.
  8. Preview and selectively apply the SSP overlay in Workshop.
  9. Optionally inspect the read-only SSP round-trip audit in Control Center observability.
  10. Refresh actions, evidence follow-up, and reports.

v0.4 — Read-only action and blocker overview

Planned capabilities

  • Consume l2g_workshop_action_summary_v1 as a local, read-only package.
  • Summarize overdue evidence, provider holds, unresolved scope questions, workbook blockers, owner/status distribution, and snapshot age.
  • Show source identity, generator version, trust labels, and compatibility before displaying counts.
  • Keep Workshop authoritative; edits, closures, reassignment, and narrative changes remain in Workshop.
  • Store only summary-package metadata unless the operator is actively viewing the selected file.

Acceptance boundaries

  • No second action database and no write-back from Control Center.
  • No automatic readiness, Met/Not Met, evidence-sufficiency, or certification conclusions.
  • Clear stale-snapshot and unsupported-version warnings.
  • Light/dark, keyboard, offline, static-security, and regression coverage.
  • Persistent roadmap updated to v0.4 with a fresh 10-release forward window.
